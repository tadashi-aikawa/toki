# Release Command

GitHubリポジトリのリリース作業を自動化します。プロダクト固有の情報(手順の詳細・投稿文の実値)はこのコマンドに書かず、下記の参照先から取得します。

## 手順の参照先

- **owleryメンバーとして作業している場合**: owlery Vaultの共有ノート `~/work/owlery/shared/knowledge/Homebrew + semantic-release型プロダクトのリリース手順.md` を単一情報源とし、上からなぞって実行する
- **それ以外のセッション**: 対象リポジトリのREADME.mdのリリース手順に従う

## 実行手順(骨組み)

1. **ローカルでpushされていないコミットがないことを確認**
   - pushされていないコミットがあれば中断

2. **CIが成功していることを確認**
   - `gh workflow` コマンドを使用する
   - GitHub ActionsのTestsワークフローが失敗していたら中断する

3. **リリース作業実行**
   - 上記「手順の参照先」に従って実行

4. **リリース完了確認**
   - GitHubリリースページを定期的に確認
   - 新しいリリースが作成されたことを確認

5. **投稿準備(外部通知の方針が定義されているプロダクトのみ)**
   - プロダクトの通知方針に従い、Bluesky / Slack等の投稿文を整形して出力する(投稿はユーザーが手動で行う)
   - プロダクト名・アップデートコマンド・Slack絵文字などの実値は、対象リポジトリのREADMEまたはプロダクトの決定事項から取得する。他プロダクトの文言を流用しない
   - 通知方針が定義されていないプロダクトではこのステップをスキップする

6. **リポジトリの最新化**
   - `git pull` でremoteの変更点を取り込む

## 必要な設定

- GitHub CLIが認証済みであること
- `gh` が403になる場合は環境変数のトークンが優先されている可能性が高い: `env -u GITHUB_TOKEN -u GH_TOKEN gh ...` で回避(詳細はowlery Vaultの「タダシ環境のgh CLIはGITHUB_TOKENが優先されて403になる」)

## 使用方法

```
/release
```
